Network Segmentation & Troubleshooting Lab

A live home network lab using OPNsense, managed switching, Cisco wireless infrastructure, Pi-hole DNS and monitoring to separate trusted, IoT, guest and management traffic while keeping useful services working.

Featured OPNsense VLANs Cisco WLC Pi-hole Monitoring

TL;DR

This project demonstrates practical network design and troubleshooting: VLAN segmentation, firewall rule design, controlled cross-VLAN access, wireless SSID mapping, DNS filtering, monitoring, and documentation of real-world trade-offs using mixed home-lab hardware.

What this demonstrates

VLAN design across trusted, IoT, guest and management networks.
OPNsense firewalling, DHCP, DNS and inter-VLAN routing decisions.
Cisco wireless controller SSID-to-VLAN mapping and AP deployment.
Structured troubleshooting using logs, client tests and monitoring evidence.

Overview

The lab is built around a real home network rather than a simulated-only topology. The current baseline uses a Virgin Hub in modem mode feeding an HP t730 running OPNsense, then a Zyxel GS1920-24v1 switch, a Cisco SG300, a Cisco 2504 WLC and wired Cisco access points.

The goal is not to make the network unnecessarily complex. The goal is to create realistic support and infrastructure scenarios: client isolation, broken DNS, firewall rule order, multicast/media discovery, switch trunking, management access and monitoring visibility.

Objective

  • • Separate trusted, IoT, guest and management traffic using VLANs and firewall policy.
  • • Keep the network usable for normal home devices while reducing unnecessary lateral access.
  • • Use the lab to practise methodical troubleshooting rather than changing settings blindly.
  • • Build evidence suitable for support, infrastructure, networking and security-adjacent roles.

Build / implementation

OPNsense acts as the central Layer 3 router and firewall. The switches carry VLANs at Layer 2, while wireless SSIDs map to the correct VLANs through the Cisco WLC. DNS filtering and visibility are provided through Pi-hole, with monitoring handled by services running on a Debian host.

Troubleshooting & decisions

  • • Moved routing and policy from OpenWrt to OPNsense to give the lab a stronger firewall and observability baseline.
  • • Kept switching at Layer 2 so firewall policy remains centralised and easier to reason about.
  • • Validated guest isolation by allowing required DNS/internet behaviour without allowing internal reachability.
  • • Solved Chromecast and media usability by allowing controlled access rather than flattening the IoT network.
  • • Treated older hardware as a design constraint rather than pretending the lab is a perfect greenfield build.

Testing & validation

  • • Confirmed DHCP, gateway and DNS behaviour per VLAN.
  • • Tested that guest clients can reach the internet but not internal networks.
  • • Tested Trusted-to-IoT access where required for media control, while preventing broad IoT-to-Trusted initiation.
  • • Verified Cisco WLC SSID mappings for Trusted, IoT and Guest wireless networks.
  • • Used Pi-hole, firewall logs and monitoring dashboards to check behaviour instead of relying on assumptions.

Outcome

The lab is now a stable, usable segmented network with working guest isolation, separated IoT devices, controlled media access, DNS filtering and monitoring. It also gives me a realistic environment for future documentation, diagrams, dashboard screenshots and deeper technical side pages.

What I learned

  • • Segmentation only matters when firewall rules, DNS, DHCP and real client workflows are tested together.
  • • Some “simple” home-network features, such as casting, expose genuinely useful troubleshooting problems.
  • • Clear documentation is part of the engineering work, not an afterthought.
